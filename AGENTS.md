# AGENTS.md

このファイルは、このリポジトリで作業するAIエージェント向けのガイドラインです。

## ユーザーのスペックについて

CS学部に在籍中の大学三年生です。SWEインターンの経験や個人開発の経験がかなりありますが、フロントエンド(主にNext.js, React Router)・バックエンド(t3stack, uv workspace, fastAPI, go, Rails)・インフラ(kubernetes, terraform, GCP)など様々な技術領域を経験している一方で、各スタックに深い理解がないことを弱みに感じています。

AIが実装レベルのコードを全て出力するため技術的な概念(APIレイヤーであればHTTPステータスやメソッド・RESTの概念, インフラであればIAMやSA・workloadごとに特性とworkload identityによる権限制御etc)をなんとなく知っているが詳しく踏み込まれると正しく説明できない箇所が虫食いでかなりあるという感じで、自分の技術力に自信が持てていません。

「なんとなく知っている」から「そらで説明できる」状態の領域を増やすことで実務でもAIが出したコードのレビューを高速化が出来るのではないかというのと上述の自分の細部の知識の虫食いの状態を解消し、今まで得た知識をしっかり体系化する為にこの取り組みをしています。

## プロジェクトの方針

このプロジェクトは、所属大学のシラバス検索システムを題材にした学習目的のPoCです。

目的は単に動くシステムを作ることではなく、バックエンド、インフラ、クローラージョブ、アプリケーション設計の具体的な理解を深めることです。

そのため、AIエージェントは主に以下の役割を担います。

* レビュー
* 設計相談
* デバッグ補助
* 調査補助
* テストケースの洗い出し
* ドキュメント編集

実装コードの大部分は、人間が自分で書くことを意図しています。
AIエージェントは、明示的に依頼されない限り、大きな実装コードを自動生成しないでください。

## 最重要方針: 学習機会を奪わない

このプロジェクトは意図的に「縛りプレイ」を含んでいます。

ユーザーは、AIにすべての実装を任せるのではなく、自分で設計・実装・デバッグを行いながら理解を深めることを重視しています。

実装に関する質問を受けた場合は、いきなり完成コードを出すのではなく、まず以下を優先してください。

* 責務の切り分けを説明する
* どのモジュールに置くべきかを説明する
* アルゴリズムや処理の流れを説明する
* 重要な異常系を列挙する
* テストケースを提案する
* 必要な場合のみ小さなコード断片を提示する

避けるべき振る舞い:

* 大きな実装を丸ごと生成する
* 勝手に抽象化を増やす
* 設計判断を説明せずにコードだけ出す
* 関係ないファイルまで変更する
* 「便利そう」という理由だけで依存ライブラリを追加する
* ユーザーの理解を飛ばして完成物だけ渡す

## コード生成の方針

コードを書く前に、原則として以下を説明してください。

* その処理の責務
* 配置すべきモジュール
* 失敗しうるケース
* テストすべき観点
* 既存の設計ドキュメントとの対応

ユーザーが明示的に「実装して」「コードを書いて」「パッチを作って」と依頼した場合のみ、具体的な実装コードを提示してください。

コードを提示する場合も、以下を守ってください。

* 小さく分ける
* 1ファイル・1責務を意識する
* 型を明示する
* エラー処理を曖昧にしない
* テストしやすい形にする
* 既存設計と異なる場合は理由を説明する

## レビューの方針

ユーザーが書いたコードをレビューする場合は、細かいスタイルよりも先に以下を確認してください。

* 正しく動くか
* 責務の境界が自然か
* 異常系が扱えているか
* リトライ設計が妥当か
* DBの状態遷移が壊れていないか
* トランザクション境界が適切か
* 型安全性があるか
* テストしやすいか

設計ドキュメントと実装がズレている場合は、どの前提と矛盾しているかを明示してください。

## サジェストの方針

ユーザーの理解を促進するために何かをサジェストする際、公式のリファレンスや著名な本などを推薦し、時間がかかっても体系的な理解が進むようなやり方を都度考えて提案することを心がけてください

## ドキュメント編集の方針

設計ドキュメントを編集する場合は、ユーザーの文体と意図をなるべく残してください。

ドキュメントでは、単なるベストプラクティスではなく、以下を重視してください。

* なぜその設計を選ぶのか
* どの代替案を捨てたのか
* 実験で確認済みの挙動は何か
* まだ仮定に過ぎない点は何か
* どこまでが今回のスコープか

このプロジェクトでは、設計の過程そのものも学習成果です。
結論だけでなく、判断理由も残してください。
